================================================================================
DASH — SEO & ARCHITECTURE PLAN
================================================================================

GOAL
----
Drive organic traffic by ranking for queries like "free online file converter",
"pdf merger online", "pomodoro timer online", etc. Each tool gets its own URL
that Google can crawl, index, and rank independently.


================================================================================
1. ARCHITECTURE: SPA → Vite SSG
================================================================================

Current state: Vite SPA — one index.html shell, React renders everything
client-side. Google gets an empty div when it crawls any URL.

Target state: Vite SSG (package: vite-ssg) — at build time, Node renders each
route to real HTML. Google crawls /file-converter and gets actual content
immediately. React still hydrates so the tool is fully interactive.

What changes in the codebase:
  - Add React Router (already needed anyway for URL routing)
  - Entry point (main.jsx) changes slightly — exports routes instead of calling
    createRoot directly; vite-ssg handles the bootstrapping
  - Add react-helmet-async (or vite-ssg's built-in head() API) for per-route
    <title>, <meta description>, and OG tags
  - Each tool route gets a static content section (see section 3)
  - sitemap.xml generated at build time
  - robots.txt added to public/

What does NOT change:
  - All existing tool components (zero changes to tools themselves)
  - Tailwind, react-grid-layout, Supabase auth — all unchanged
  - Vercel deployment — Vercel serves static files natively, zero config needed

3. WHAT GOES ON A TOOL PAGE
================================================================================

Each /[tool-id] page needs three things for SEO:

A. <head> metadata (invisible to users, critical for Google)
   - <title>: "Free Online [Tool Name] – Dash"
     Example: "Free Online File Converter – Dash"
   - <meta name="description">: 1–2 sentence summary with the primary keyword
     Example: "Convert files between formats instantly in your browser. No
     uploads, no installs — just drag, drop, and convert."
   - <meta property="og:title"> and <meta property="og:description"> for
     social sharing previews
   - JSON-LD structured data (WebApplication schema) — invisible, in <head>,
     tells Google exactly what the page is

B. Minimal visible content — keeping the UI clean
   The goal is NOT a big content block. For a tool/app page, meta tags + JSON-LD
   carry the bulk of the SEO weight. What little visible text is needed:

   Option 1 (minimal): A single muted line above the grid acting as a page label
     "File Converter  ·  Convert files between formats in your browser"
     Small, subdued, styled like a page subtitle — not a marketing paragraph.

   Option 2 (none extra): The tool card itself already contains the tool name and
     description. That's real, indexable content. Combined with good meta tags
     it's often enough for medium-competition keywords.

   What NOT to do:
   - display: none, visibility: hidden, white text on white — Google penalizes
     this explicitly, risk of de-indexing
   - sr-only CSS tricks for keyword content — gray area, risky if overused

   The tool card's existing name + description is legitimate content. Lean into
   it. The JSON-LD carries the structural weight invisibly.

C. The tool itself pre-loaded on the dashboard
   When you land on /file-converter, the dashboard opens with the File Converter
   card already placed on the grid. User can immediately use it and add more
   tools from the + button.

================================================================================
5. GUIDES (FUTURE — PHASE 2)
================================================================================

Separate tab in the Dash navbar. Purpose: rank for informational queries and
build domain authority, which helps ALL tool pages rank better.

FORMAT DECISION: Guides as Dash cards, not traditional blog posts.
  - Each guide is a large card on the grid (e.g. 10×18 cells), internally
    scrollable, with proper <h1>/<h2>/<p> tags inside the card component
  - The relevant tool card sits alongside it on the same grid
    Example: "How to merge PDFs" guide card + PDF Merger tool card side-by-side
  - This is unique — no other site presents guides this way
  - Conversion funnel is built in: reader sees the product in action while
    reading, can use the tool immediately without leaving the page
  - Google doesn't care about visual layout, only that the text is in the
    pre-rendered HTML — which it will be via SSG

  Why this beats a traditional blog:
  - Differentiated — memorable, matches the brand
  - Contextual adjacency: guide about PDF merging → PDF Merger tool right there
  - No "click here to try our tool" CTA needed — it's already visible
  - Still ranks the same as a blog post if the content is substantive

  The one thing to ensure: the full guide text is in the SSG-rendered HTML, not
  lazy-loaded or hidden behind a client-side tab. Internal scroll is fine —
  Google reads content below the fold.

Planned guide topics (from ideas.txt + additions):
  - How to protect your data online                   → pairs with: notepad
  - Password management best practices                → pairs with: uuid-generator
  - What is a PDF and how do I work with them?        → pairs with: pdf-merger
  - How to compress files before emailing             → pairs with: file-converter
  - What is a UUID and why do developers use them?    → pairs with: uuid-generator
  - How to use the Pomodoro Technique                 → pairs with: pomodoro-timer
  - Understanding binary and number bases             → pairs with: number-converter

Each guide targets a long-tail keyword. The paired tool card is pre-loaded on
the grid when you visit the guide URL.

Domain authority benefit: guides bring in traffic that has nothing to do with
the tools. Links to the tools from within guide content pass authority to tool
pages, helping them rank higher over time.


================================================================================
6. TECHNICAL SEO CHECKLIST
================================================================================

sitemap.xml
  - Auto-generated at build time listing all tool URLs + guides
  - Submitted to Google Search Console after launch
  - Resubmitted whenever new tools are added

robots.txt (in public/)
  - Allow all crawlers
  - Point to sitemap: Sitemap: https://dash.tech/sitemap.xml

Page speed (Core Web Vitals — Google ranking factor)
  - FFmpeg is large (~30MB WASM). Already lazy-loaded inside tools, only runs
    when the tool is actually used. Do NOT load it at page level.
  - Code-split tool bundles: each tool loads its own JS only when added to grid
    (currently true since they're dynamic imports via the registry)
  - Avoid loading all 14+ tool bundles on initial page load

Canonical tags
  - Each page should have <link rel="canonical" href="https://dash.tech/[slug]">
  - Prevents duplicate content issues if the app is ever accessible at multiple
    URLs (www vs non-www, etc.)

Structured data (optional, nice to have)
  - JSON-LD WebApplication schema on each tool page
  - Helps Google understand what the page is (a tool/app) vs a blog post

================================================================================
9. ADDITIONAL THOUGHTS & OPPORTUNITIES
================================================================================

On differentiation:
  Dash's strongest angle is that it's the only tool site that looks and feels
  like a real product — not a sea of AdSense. That visual cleanliness is itself
  a reason people share it and return to it. Don't compromise it for SEO.
  The architecture above lets you win on SEO without touching the UI philosophy.

On the guide-as-card format:
  This is genuinely novel. If it works well, it's a story worth telling —
  YouTube video, Reddit posts, HN post. "I built a tool site where the guide
  and the tool are literally on the same page." That kind of originality gets
  organic links, which are the best SEO signal of all.

On tool selection strategy:
  High-volume keywords ("file converter", "PDF merger") are competitive but
  winnable with enough domain authority. Start by targeting easier ones:
  - "browser notepad no account"
  - "tip calculator split bill"
  - "uuid v4 generator online"
  Then build toward the harder ones as the domain ages and gains authority.
  New domains typically take 6–12 months to rank for anything competitive.

On privacy as a differentiator:
  "No uploads — everything runs in your browser" is a real advantage over
  competitors that send files to a server. This should be in every meta
  description and mentioned in guides. It's a trust signal AND a feature.
  Privacy-conscious users actively search for "client-side" or "no upload"
  tools. Those are low-competition long-tail keywords worth targeting.

On future tool ideas that map well to SEO:
  - Image converter/compressor ("compress image online")
  - Color picker / palette generator ("color picker hex")
  - QR code generator ("qr code generator free")
  - Password generator ("strong password generator")
  - JSON formatter ("json formatter online")
  - Markdown editor ("markdown editor online")
  - Base64 encoder/decoder ("base64 encode online")
  Each of these has clear search demand and can be built as self-contained cards.

On the subscription model:
  The guide content + privacy angle + clean UX creates a natural argument for
  a paid tier: "Dash Pro — save unlimited dashboards, sync across devices,
  early access to new tools." Free tier stays fully functional, Pro adds
  persistence and convenience. No paywalling tools that are free elsewhere.
